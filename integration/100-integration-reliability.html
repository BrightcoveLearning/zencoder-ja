    100% 統合信頼性

    このトピックでは、Zencoder に接続できない理由と、信頼性の高い統合を実現する方法について説明します。

    概要

    Zencoderは、ほとんどのお客様にとって不可欠なソフトウェア依存関係です。また、100％の稼働時間を目指していますが、Zencoderに接続できない場合があります。

    これが発生してZencoderがダウンすると、アプリケーションは通常Zencoderから「503Service Unavailable」応答を受け取りますが、別のエラー（500など）が発生する可能性があります。APIレート制限を超えた場合は、「403レート制限を超えました」という応答が返されます。

    良いニュース：ビデオエンコーディングは非同期プロセスであるため、ダウンタイムや可用性に関連する問題が発生しないようにアプリケーションを構築できます。これを行う場合、最悪のシナリオは、ジョブに少し時間がかかることです。ただし、エラーは発生しません。これを行うことを強くお勧めします。

    もっと強く言えば、信頼性が気になる場合は、このアプローチに従って統合する必要があります -Zencoderの場合、または統合する重要なAPIの場合。

    • 当社のサービスは、アップストリームプロバイダー（Amazon Web Servicesなど）の問題の影響を受ける可能性があります
    • 一時的なダウンタイムを必要とするシステムメンテナンスを実行する必要がある場合があります
    • あなたはあなたを超えましたAPIレート制限
    • 等

    信頼性の高いアプリ統合

    1. 含めるセカンダリURLプライマリロケーションへのアップロードが失敗した場合のバックアップとして。
    2. Zencoderから失敗した応答コードを受け取った場合（基本的には、200または201以外のもの）仕事を失敗させないでください。503の応答コードは、ビデオを処理できないことを意味するものではありません。Zencoderが一時的に利用できないことを意味します。
    3. あなたが得る場合接続エラー Zencoderに接続しようとするときも、同じことを行います。
    4. 同様に、 APIリクエストをタイムアウトでラップする。30秒のタイムアウトをお勧めします。 Zencoderは通常、1秒未満で応答するため、通常は30秒で十分です。
    5. これらの3つのケースすべてで、応答コードが失敗した場合、接続できない場合、またはAPIリクエストがタイムアウトした場合。ジョブに「保留中」のフラグを付けます。
    6. 定期的に、 '保留中'状態のジョブを再送信します。たとえば、cronを使用してこれを毎分行うことができます。

    ジョブが再送信されると、すべてが通常のように動作します。このように、失敗したジョブの送信は、アプリケーションやユーザーに問題を引き起こすのではなく、ジョブに少し時間がかかるだけです。

    擬似コード

    OK、これは擬似コードではなく、Rubyです。しかし、Rubyは非常に読みやすいです。

    1. これらの列を含むVideosテーブルを想像してみてください。（ZencoderジョブIDとZencoder出力ファイルIDを格納する列など、明らかに多くの情報があります。） 
      create_table :videos do |t|
t.string  :state
t.integer :lock_version
t.index   :state
end
    2. ビデオには、次の状態のステートマシンが含まれている必要があります。
      • 保留中（まだZencoderに送信されていません）
      • 提出（現在Zencoderに提出）
      • トランスコーディング（Zencoderへの送信に成功）
      • 終了しました（Zencoderがトランスコーディングを終了し、ジョブが完了しました）
      • 失敗しました（Zencoderはビデオをトランスコードできませんでした）
    3. 新しいビデオが取り込まれたら、ビデオを「送信」状態で保存し、バックグラウンドジョブをトリガーしてビデオをZencoderに送信します。 
      # got a new video!
video = Video.new(params)
video.state = "submitting"
video.save!
submit_to_zencoder(video)

      あなたは本当にsubmit_to_zencoderメソッドをバックグラウンドにする必要があります。Rubyでは、DelayedJobを使用すると、次のようになります。

      delay.submit_to_zencoder(video)

      しかし、私たちは私たちに固執しますsubmit_to_zencoder(video)例の目的のための方法。

    4. ザ・submit_to_zencoder関数は次のようになります。これは、バックグラウンドで非同期に実行する必要があります。 
      def submit_to_zencoder(video)
begin
response = Zencoder::Job.create(attributes, :timeout => 30_000)
if response.code == 201
video.state = "transcoding"
else
video.state = "pending"
end

video.save!

# Rescue any connection error. Our plugin abstracts these as
# Zencoder::HTTPError.
#
# If you're not using the Zencoder plugin, this includes things
# like Errno::ECONNRESET, Errno::ETIMEDOUT, Errno::ECONNREFUSED,
# Errno::EHOSTDOWN, and SocketError.

rescue Timeout::Error, Zencoder::HTTPError
video.state = "pending"
video.save!
end
end
    5. 時々（たとえば毎分）、「保留中」の状態にあるジョブを再送信しようとします。 
      def resubmit_pending_jobs
Video.where(:state => "pending").find_each do |video|
begin
video.state = "submitting"
video.save!

submit_to_zencoder(video)
rescue ActiveRecord::StaleObjectError
end
end
end

      また、ビデオテーブルに「lock_version」列を追加することで、楽観的ロックを導入します。これは、レコードがVideo.findクエリとvideo.save、Zencoderにジョブを送信しません。これにより、ジョブが誤って2回Zencoderに送信されるのを防ぐことができます。同じことを達成するために、悲観的またはデータベースロックまたは他のロック方法を使用することができます。

      とても簡単です…

      すべてを考慮すると、これはZencoderとアプリケーション間の100％の統合信頼性を確保するための非常に単純なアプローチです。これは、単純にジョブを送信するよりもいくつかの手順です。ただし、偶発的なタイムアウト、Zencoderでの予期しないダウンタイム、定期メンテナンスなど、何が起こってもアプリは確実に実行されます。
